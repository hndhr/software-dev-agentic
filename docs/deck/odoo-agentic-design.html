Agentic System
for Odoo Development

A single-repo Claude Code toolkit that covers the full SDLC for Odoo — from requirement analysis to implementation and testing. Four personas. Seven agents. Six orchestrator skills.

analyst designer developer tester
Context

Odoo is framework-heavy. Errors are cryptic. Patterns repeat.

Without agentic
With agentic
Every change requires tracing the _inherit chain manually before writing a single line.
Planner traces the chain first — worker writes with full context, zero guesswork.
4 artifact types always move together — model, view, security, manifest. Easy to miss one.
Worker calls the right P skills in sequence. Manifest update is enforced every time.
Odoo errors are cryptic — AccessError, ORM constraint, broken XPath. Hard to categorize fast.
Debug worker loads a decision tree, categorizes first, traces from the right starting point.
New module vs extend existing — a design decision made inconsistently, often too late.
Designer persona resolves this upfront. Developer picks it up from module-design.md.

Architecture · Components

Three components. Each with one clear responsibility.

Reference — The Knowledge

Markdown files agents load whole. Scoped tightly — one file per artifact type or concern. Each file answers one question a specific agent needs to ask.

model-impl.md · view-impl.md
security-impl.md · debugger-impl.md
testing-impl.md · controller-impl.md
If a file grows beyond what one agent needs — split it. Don't annotate it.
Skill — The Hands

Procedural instructions that run in the caller's session. Two types: Orchestrator (O) — user entry points that drive the full workflow. Procedure (P) — thin create steps called by workers.

O — user entry
/analyst-analyze-requirement
/developer-build-feature

P — worker-only
odoo-model-create/SKILL.md
odoo-manifest-update/SKILL.md
Agent — The Brain

Isolated reasoning in its own context window. Loads reference docs, makes decisions, returns structured output. Context cost stays out of the main session.

Strategist — routes + synthesizes
Planner — explores, read-only
Worker — executes, writes files
Agent complexity is matched to workflow complexity — not every persona needs a strategist.
Skills reach the codebase. Agents reason about it. Reference docs inform both. Each component has one job and does nothing else.
Architecture · Persona Roster

Four personas. Agent structure matched to workflow complexity.

Persona SDLC Phase O Skills Agents Why this structure
analyst Requirement 1 worker only Pure reasoning — no code exploration needed
designer Design 1 worker only Light module exploration + design in one pass
developer Implementation · Debug 3 strategist + planner + 2 workers Inheritance chain demands pre-exploration; build vs debug are distinct pipelines
tester Testing 1 worker only Write task with clear reference patterns
Developer earns the multi-agent pipeline — the _inherit chain problem means you can't start writing without exploring first. The other three personas don't have that constraint.
Architecture · Orchestrator Skills — I/O Contracts

Six O skills. Each with explicit input and output.

Skill Input Output Pipeline
/analyst-analyze-requirement Raw requirement text; existing requirements.md (optional) requirements.md → analyst-worker
/designer-plan-module requirements.md (optional); module path (optional) module-design.md → designer-worker
/developer-build-feature module-design.md (optional); feature name Source files — model, view, security, manifest → strategist → planner → feature-worker
/developer-quick-edit Scoped instruction (required); file path (optional) Targeted edit to existing source file(s) → feature-worker directly
/developer-debug Error message + stack trace (required); file path (optional) Fixed file(s) + root cause explanation → debug-worker
/tester-write-tests Module path (required); plan.md (optional) tests/test_<feature>.py + manifest updated → tester-worker
analyst and designer handle new + update in one skill — they detect whether the output file exists and create or revise accordingly. developer has 3 skills — build, quick-edit, and debug are genuinely different pipelines.

Build Priority

Start with reference docs. Everything else depends on them.

01
Reference docs — model · view · security · module-structure
Every agent loads at least one of these. Build them first or nothing else works correctly.
02
developer implementation pipeline + /developer-build-feature
strategist + planner + feature-worker + all P skills. Highest daily use.
03
developer-debug-worker + debugger-impl.md + /developer-debug
High value — Odoo errors are painful without a structured decision tree.
04
designer-worker + /designer-plan-module
Optional but improves implementation quality. Skippable for small changes.
05
analyst-worker + /analyst-analyze-requirement
Useful for complex features. Skippable when scope is already clear.
06
tester-worker + testing-impl.md + /tester-write-tests
v2 — blocked on testing-impl.md being authored first. Without it, worker will produce structurally incorrect tests.
