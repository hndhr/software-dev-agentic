The Problem

Building features across platforms creates invisible drift

  • An engineer picks up a ticket. The first hour is spent reading code — understanding structure, conventions, where things go. Not building.
  • Three platforms. Three codebases. The same architecture decisions made differently, drifting over time, slowing down every engineer who touches them next.
  • Agents in Claude Code promised to help — but without a shared system, every project grew its own. The problems got worse, not better.

This is not a people problem. It is a system problem.

The Architecture

Three problems. Three architectural answers.

Problem
Solution
Bloated context — agents and CLAUDE.md carry everything
Tiered knowledge architecture — lean agents, reference docs loaded on demand
Knowledge not isolated — full reference file read on every call
Grep over Read — fetch only the section needed, only when needed
No shared agents — three projects, three copies, constant drift
Git submodule — one file, symlinked to every project, updated once
Together with prompt caching, these three decisions reduced AI compute cost by 85% in four calendar days.

Solution — Prompt Caching

    
Pay once. Reuse many times.

    

      

        

          
Cached — paid once per session

          Agent system prompt (agent file content)

          Preloaded skills (SKILL.md files)

          CLAUDE.md (universal project rules)
        

        
↕ cache boundary

        

          
Not cached — changes each call

          Current task / user message

          Grepped reference snippets

          Worker outputs from prior phases
        

      

      

        
    
          
  • Agent files are stable — cache hit on every subsequent worker spawn in a session.
    • 
          
  • Preloaded skills injected once at startup, cached for the whole session.
    • 
          
  • CLAUDE.md is short (~1 page) — small stable prefix, high cache efficiency.
    • 
          
  • Compounds across workers — in a 5-worker feature build, the stable prefix is paid for once. Workers 2–5 reuse it from cache.
    • 
          
  • Context relay — trigger skills read context.md and state.json in the same session (cache read: $0.30/MTok) and pass them inline. The orchestrator skips cold pre-flight reads entirely ($3.75/MTok avoided).
    • 
        

      

    

    

      Lean agents + Grep-first + caching = the cost curve bends down
      as usage goes up, not up.

What We Built

    

      Not a chatbot.
Not autocomplete.
A hierarchy of specialized AI agents.
    

    

    
    
      
  • Each agent is an expert in exactly one layer of the codebase — and nothing else.

    • 
      
  • The engineer's job shifts from writing boilerplate to reviewing output.

Architecture — Builder Persona Example

    
Builder persona. Five roles in practice.

    

      

        
Trigger Skill

        
/feature-orchestrator · /plan-feature · /backend-orchestrator

feature-orchestrator

Planner

        
feature-planner · domain-planner · data-planner · pres-planner

        

          feature-planner spawns three layer planners in parallel. Each explores its layer and returns structured findings aggregated into plan.md.

Worker

        
feature-worker (orchestrated) · domain-worker · data-worker · ui-worker (standalone)

        

          feature-worker is the orchestrated-path executor — reads the approved plan and calls skills across all CLEAN layers in sequence.


          domain-worker · data-worker · ui-worker handle single known artifacts directly, bypassing the orchestrator.
        

      

      

        
Skill

Knowledge

    
Three tiers. Each loaded only when needed.

    

      

        

          
Tier 1 — CLAUDE.md

          Universal rules. ~1 page. Loaded every turn.

          Naming conventions, architecture boundaries, quality gates.
        

        

          
Tier 2 — Agent body

          Decision logic for that agent only.

          When to act, what to check, how to handle failure.

          Not shared. Not duplicated. Scoped to one agent.
        

        

          
Tier 3 — Reference docs

          Deep architectural knowledge: layer contracts, patterns, platform examples.

          Loaded on demand via Grep — one section at a time.
        

      

      

        
    
          
  • 
            Reference docs are greppable by design — every file uses the same heading structure
            (## SectionName)
            so workers fetch exactly the section they need.
          
    • 
          
  • 
            Knowledge is never duplicated — one reference file, multiple workers reading the same section via Grep.
            Update once, every agent sees it immediately.
          
    • 
          
  • 
            Self-enforcing — an internal arch-reviewer audits every agent and skill file for convention violations before it ships to any downstream project. The system checks itself.
