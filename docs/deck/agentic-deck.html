The Problem

Building features across platforms creates invisible drift

  • An engineer picks up a ticket. The first hour is spent reading code — understanding structure, conventions, where things go. Not building.
  • Three platforms. Three codebases. The same architecture decisions made differently, drifting over time, slowing down every engineer who touches them next.
  • Agents in Claude Code promised to help — but without a shared system, every project grew its own. The problems got worse, not better.

This is not a people problem. It is a system problem.

The Architecture

Three problems. Three architectural answers.

Problem
Solution
Bloated context — agents and CLAUDE.md carry everything
Tiered knowledge architecture — lean agents, reference docs loaded on demand
Knowledge not isolated — full reference file read on every call
Grep over Read — fetch only the section needed, only when needed
No shared agents — three projects, three copies, constant drift
Git submodule — one file, symlinked to every project, updated once

Solution — Token Efficiency

Same file. A fraction of the context.

❌ Read — entire file enters context
# Returns the whole file to the agent
Read("reference/contract/builder/domain.md")

# → 200 lines added to context window
# → Agent pays for all of it, every call
# → Even when only one section is relevant
✓ Grep + targeted Read — only the section needed
# Reference doc heading carries its own line count:
# ## Entity    ← auto-updated by pre-commit hook

# Step 1: locate section, read count from heading
Grep("^## Entity",
  "reference/contract/builder/domain.md")
# → "## Entity " at line 42

# Step 2: read exactly that section
Read("reference/contract/builder/domain.md",
  offset=42, limit=15)
# → 15 lines. Dynamic. Always accurate.
reference/contract/builder/domain.md — what the file actually looks like
## Entity         ← heading + auto-maintained line count

The annotation is maintained automatically by a pre-commit hook — engineers edit the doc freely, the hook rewrites the count on every commit. No hardcoded limits. No stale counts.
Solution — Prompt Caching

Pay once. Reuse many times.

Cached — paid once per session
Agent system prompt (agent file content)
Preloaded skills (SKILL.md files)
CLAUDE.md (universal project rules)
↕ cache boundary
Not cached — changes each call
Current task / user message
Grepped reference snippets
Worker outputs from prior phases
  • Agent files are stable — cache hit on every subsequent worker spawn in a session.
  • Preloaded skills injected once at startup, cached for the whole session.
  • CLAUDE.md is short (~1 page) — small stable prefix, high cache efficiency.
  • Compounds across workers — in a 5-worker feature build, the stable prefix is paid for once. Workers 2–5 reuse it from cache.
Lean agents + Grep-first + caching = the cost curve bends down as usage goes up, not up.

What We Built

Not a chatbot.
Not autocomplete.
A hierarchy of specialized AI agents.

  • Each agent is an expert in exactly one layer of the codebase — and nothing else.
  • A Tech Lead orchestrator coordinates them in the right order, passes only what each needs, and compiles the result.
  • The engineer's job shifts from writing boilerplate to reviewing output.
Architecture

Three roles. Each knows exactly what it owns.

Orchestrator
Coordinates workers in a defined sequence. Validates each worker's output before proceeding. Never writes files directly — all writes go through workers.

Passes only what each worker needs. Stops immediately if a worker fails.
Worker
Owns exactly one responsibility. Reads reference docs on demand, executes skills, returns output. Refuses work outside its scope — names the correct worker instead.

Platform-agnostic by design. The same worker logic runs across Flutter, iOS, and Web.
Skill
Step-by-step procedure for one atomic task. Does one thing only, under 30 lines. Platform-specific — the same skill name generates different output per platform.

Swapping the platform swaps the skills. No agent changes needed.
Workers are the brains — they decide what to do. Skills are the hands — they know how to do it. This separation is what makes the system extensible across platforms without touching any agent.

Knowledge

Three tiers. Each loaded only when needed.

Tier 1 — CLAUDE.md
Universal rules. ~1 page. Loaded every turn.
Naming conventions, architecture boundaries, quality gates.
Tier 2 — Agent body
Decision logic for that agent only.
When to act, what to check, how to handle failure.
Not shared. Not duplicated. Scoped to one agent.
Tier 3 — Reference docs
Deep architectural knowledge: layer contracts, patterns, platform examples.
Loaded on demand via Grep — one section at a time.
  • Reference docs are greppable by design — every file uses the same heading structure (## SectionName) so workers fetch exactly the section they need.
  • Knowledge is never duplicated — one reference file, multiple workers reading the same section via Grep. Update once, every agent sees it immediately.
  • Self-enforcing — an internal arch-reviewer audits every agent and skill file for convention violations before it ships to any downstream project. The system checks itself.
